# 기술 스택 결정

## 결정 요약

꿈꾸민턴 대진App MVP는 `Vue 3 + Cloudflare` 기반의 모바일 우선 웹앱으로 만든다.

```text
Frontend: Vue 3 + Vite + TypeScript
State: Pinia
Routing: Vue Router
Hosting: Cloudflare Pages
API: Cloudflare Pages Functions
Session Storage: Cloudflare D1
Member Source: Google Sheets 회원명단, read-only
Sheet Automation: Google Apps Script, 기존 관리 자동화 유지
```

## 배경

이 앱은 배드민턴 모임 안에서만 쓰는 극소규모 운영 도구다. 운영자는 경기장에서 모바일로 참석자를 고르고 대진을 만들며, 모임원은 카카오톡 오픈채팅방 링크로 들어와 현재 경기와 본인 순서를 확인한다.

따라서 기술 선택의 기준은 다음 순서로 둔다.

1. 운영 비용이 거의 없어야 한다.
2. 카카오톡 인앱 브라우저, 모바일 Safari, Chrome에서 안정적으로 동작해야 한다.
3. 로그인 없이도 공개 경기판을 볼 수 있어야 한다.
4. 운영자 수정 링크와 멤버 조회 링크는 분리되어야 한다.
5. Google Sheet `회원명단`을 회원 원본으로 유지해야 한다.
6. 매칭 로직은 UI와 분리되어 테스트 가능해야 한다.

## 프론트엔드

### Vue 3

Vue 3를 사용한다. 화면은 모바일 중심의 상태 전환이 많고, 운영자용 입력 화면과 멤버용 경기판 화면이 분리된다. Vue는 컴포넌트 단위로 화면을 작게 나누기 쉽고, 개인 학습 스택으로도 적합하다.

### Vite

개발 서버와 번들러는 Vite를 사용한다. 빠른 로컬 개발과 Cloudflare Pages 정적 배포에 잘 맞는다.

### TypeScript

회원, 참석자, 경기, 라운드, 세션 타입을 명시한다. 매칭 로직이 데이터 형태에 민감하므로 TypeScript를 기본으로 쓴다.

### Pinia

운영자 화면의 세션 상태를 관리한다.

관리 대상:

- 선택된 참석자
- 코트 수
- 코트별 현재 상태
- 완료된 경기 이력
- 대기 큐
- 대기 큐 수동 조정 이력
- 대기 큐 우선권/보류 상태
- 참석자별 오늘 경기 수
- 참석자별 참여 빈도 설정
- 공유 링크 상태

### Vue Router

라우팅은 단순하게 시작한다.

```text
/admin/new
/admin/:sessionId?key=...
/board/:sessionId
```

## 백엔드와 호스팅

### Cloudflare Pages

정적 Vue 앱을 Cloudflare Pages에 배포한다. 비용이 낮고, 정적 파일 제공과 Functions를 한 프로젝트에서 다룰 수 있다.

### Cloudflare Pages Functions

MVP에서 필요한 API는 작다.

```text
GET  /api/health
GET  /api/members
POST /api/sessions
GET  /api/sessions/:id
PUT  /api/sessions/:id
```

`/api/members`는 Google Sheet CSV를 읽어서 앱에 필요한 JSON으로 변환한다. 브라우저에서 Google Sheet CSV를 직접 fetch하면 CORS 문제가 생길 수 있으므로 Cloudflare Function을 프록시로 둔다.

### Cloudflare D1

회원 DB는 만들지 않는다. D1에는 오늘 경기 세션만 저장한다.

저장 대상:

- `sessionId`
- `writeTokenHash`
- `attendees`
- `courtCount`
- `courts`
- `waitingQueue`
- `completedMatches`
- `updatedAt`
- `expiresAt`

D1을 선택하는 이유:

- 경기판 업데이트가 즉시 조회되어야 한다.
- KV보다 세션 조회/수정 모델이 명확하다.
- MVP 규모에서는 비용 부담이 거의 없다.
- 나중에 라운드 히스토리 테이블로 확장할 수 있다.

## 데이터 원본

### Google Sheet 회원명단

회원 원본은 Google Sheet `회원명단` 탭이다. 앱은 이 시트를 읽기 전용으로만 사용한다.

현재 기준 컬럼:

```text
No
가입일
회원명
급수(지역)
점수(백분위)
성별
운영진(Y/N)
면제(Y/N)
비고
```

앱에서 공개 경기판으로 내려보내지 않을 정보:

- 가입일
- 운영진 여부
- 면제 여부
- 비고
- 점수(백분위)

점수는 매칭 알고리즘 내부에서만 사용한다.

### Google Apps Script

기존 출석/회원 관리 자동화는 `apps-script/sheet-automation`에서 유지한다.

주의:

- Apps Script는 Node 10 + clasp 2.x 기준으로 관리한다.
- 웹앱은 Node 22 + npm 11 기준으로 관리한다.
- 두 개발 환경은 의도적으로 분리한다.

## 매칭 엔진

매칭 로직은 UI와 분리된 순수 함수로 작성한다.

주요 입력:

- 참석자 목록
- 코트 수
- 코트별 현재 상태
- 오늘 완료 경기 이력
- 대기 큐 수동 조정 이력
- 대기 큐 우선권/보류 상태
- 참석자별 참여 빈도 설정
- 시드 값

주요 출력:

- 코트별 현재 상태와 경기 목록
- 선입선출 대기 큐
- 완료 경기 메타데이터

MVP 매칭 원칙:

- 한 경기에는 정확히 4명이 필요하다.
- `gamesPerRound = min(courtCount, floor(attendeeCount / 4))`
- MVP에서는 경기 종류 모드를 두지 않고 전체 참석자를 통합 매칭한다.
- 앱은 현실의 경기 시작/종료를 추측하지 않는다. 코트 상태는 운영자가 누른 `시작`/`종료` 버튼만 기준으로 바뀐다.
- 코트는 현재 배정된 경기와 상태를 가진다.
- 코트 상태는 `비어있음`, `배정됨`, `진행중`으로 나눈다.
- 대기자는 특정 코트나 미래 경기 조에 미리 배정하지 않고 하나의 개인 대기 큐로 관리한다.
- 운영자가 특정 코트의 `종료`를 누르면 해당 경기를 완료 처리하고, 대기 큐 앞쪽 사람들을 우선해 다음 4명을 그 코트에 `배정됨` 상태로 넣는다.
- 오래 쉰 사람을 우선한다.
- 직전 라운드에 뛴 사람은 우선순위를 낮춘다.
- `진행중` 코트 배정은 잠근다. 운영자는 이미 시작된 경기의 선수를 이동하거나 교체할 수 없다.
- 운영자는 대기 큐 순서를 터치 드래그로 바꿀 수 있어야 한다.
- 운영자는 대기 큐 안의 사람에게 우선권 또는 보류 상태를 줄 수 있어야 한다.
- 운영자는 참석자별 참여 빈도를 `높음`, `일반`, `낮음`으로 설정할 수 있어야 한다.
- 오늘 완료한 경기 수는 참석자 목록과 운영 화면에서 확인할 수 있어야 한다.
- 대기자는 특정 코트에 미리 배정되지 않으므로 “2코트의 사람을 1코트로 이동” 같은 조작은 존재하지 않는다.
- 파트너 반복을 상대 반복보다 먼저 줄인다.
- 실력 차이는 강한 감점으로 처리하되, 사람이 부족하면 완화한다.

## 코트 상태 흐름

```text
비어있음
  -> 대기 큐에서 4명 자동 배정
배정됨
  -> 운영자가 시작 누름
진행중
  -> 운영자가 종료 누름
완료 기록
  -> 같은 코트에 대기 큐 기준 다음 경기 자동 배정
```

### 비어있음

코트에 현재 배정된 경기가 없는 상태다. 대기 큐에 4명 이상 있으면 운영자가 수동 배정하거나, 직전 경기 종료 후 자동 배정된다.

### 배정됨

코트에 다음 경기 4명이 배정되었지만, 운영자가 아직 `시작`을 누르지 않은 상태다. 이 상태에서는 운영자가 배정을 취소하거나 다시 계산할 수 있다.

### 진행중

운영자가 `시작`을 누른 상태다. 이 상태부터는 실제 경기가 진행 중이라고 보고 앱에서 배정을 잠근다.

### 종료

운영자가 `종료`를 누르면 앱은 해당 경기를 완료 처리한다. 이후 같은 코트에 대기 큐 기준 다음 경기를 자동 배정한다. 대기 큐에 4명 미만이면 코트는 `비어있음`으로 남는다.

## 진행 중 경기 잠금 정책

운영자가 `시작`을 누른 코트의 현재 경기는 진행 중인 게임으로 간주한다. 진행 중인 게임은 운영자 화면에서도 수정할 수 없다.

허용되는 코트 액션:

- 시작: `배정됨` 상태의 경기를 `진행중`으로 바꾼다.
- 종료: `진행중` 상태의 경기를 완료하고, 대기 큐 기준으로 다음 경기 4명을 구성한다.
- 배정 취소/재배정: `배정됨` 상태에서만 허용한다.
- 결과 복사/공유: 현재 배정 정보를 읽기 전용으로 보여준다.

허용되지 않는 액션:

- 진행 중인 코트에서 특정 선수를 빼기
- 진행 중인 코트의 선수를 다른 코트로 이동하기
- 다른 진행 중 코트에서 선수를 가져와 빈 슬롯 채우기
- 진행 중 경기의 팀 조합을 재계산하기

이 정책을 두는 이유는 경기 중인 4명의 실제 플레이를 앱 조작으로 흔들지 않기 위해서다. 코트가 먼저 끝나면 운영자가 `종료`를 누르고, 그때 대기 큐에서 다음 경기를 구성한다.

## 대기 큐 수동 조정

운영자는 대기 큐에서 사람을 길게 누르거나 드래그 핸들을 잡아 위아래로 이동할 수 있다.

사용 목적:

- 실제 체육관 상황 반영
- 잠시 자리를 비운 사람 뒤로 이동
- 곧 나가야 하는 사람 앞으로 이동
- 운영자가 직접 조율한 배려 반영

추가 상태:

- 우선권: 다음 빈 코트 배정 계산에서 앞쪽 후보로 본다.
- 보류: 잠시 자리를 비운 사람을 다음 배정 후보에서 제외하거나 뒤쪽으로 보낸다.

수동 순번 변경은 매칭 점수보다 우선한다. 다만 너무 큰 실력 차이나 반복 파트너 문제가 있으면 앱은 경고 표시를 할 수 있다.

## 오늘 경기 수와 참여 빈도

운영자는 그날 참석자마다 몇 게임을 쳤는지 확인할 수 있어야 한다. 이 값은 경기 `종료` 버튼을 눌렀을 때 해당 코트의 4명에게 1게임씩 증가한다.

참여 빈도 설정은 원본 `회원명단`에 저장하지 않고, 오늘 세션 안에서만 관리한다. 컨디션이나 일정에 따라 같은 사람도 매번 선호가 달라질 수 있기 때문이다.

```text
높음: 게임을 더 자주 치고 싶거나 체력이 충분한 참석자
일반: 기본값
낮음: 너무 많은 게임이 부담되는 참석자
```

매칭 계산에서는 오늘 경기 수를 그대로 비교하지 않고 빈도 가중치로 보정한다.

```text
높음: targetWeight = 1.2
일반: targetWeight = 1.0
낮음: targetWeight = 0.8

effectiveGames = gamesPlayed / targetWeight
```

`effectiveGames`가 낮은 참석자일수록 다음 배정 후보에서 유리하다. 예를 들어 2게임을 친 참석자라도 `높음`이면 1.67게임처럼 계산되고, `낮음`이면 2.5게임처럼 계산된다.

이 설정은 절대 규칙이 아니라 우선순위 보정이다. 참석자가 부족하면 `낮음` 참석자도 배정될 수 있고, `높음` 참석자도 직전 경기 연속 출전이나 실력 차이 문제가 있으면 감점될 수 있다.

실력 점수 정책:

```text
좋음: 20점 이내
허용: 30점 이내
부족 시: 30점 초과도 허용하되 큰 감점
```

## 보안과 권한

MVP에서는 로그인하지 않는다.

대신 링크를 분리한다.

```text
멤버 조회 링크: /board/:sessionId
운영자 수정 링크: /admin/:sessionId?key=writeToken
```

운영자 링크의 `writeToken`은 세션 생성 시 발급하고, 서버에는 hash만 저장한다.

공개 경기판은 읽기 전용이다.

## 실시간성

WebSocket은 MVP에서 쓰지 않는다. 카카오톡 인앱 브라우저와 모바일 환경의 안정성을 우선한다.

멤버 경기판은 3~5초 간격으로 폴링한다.

응답에는 `updatedAt` 또는 `version`을 포함해 화면에 마지막 갱신 시간을 표시한다.
